# LiftGrid Systems — Database Design Brief

> **Task type:** Database design only — no SQL queries, no APIs required.

---

##  Company Overview

**LiftGrid Systems** is a fast-growing infrastructure technology company that builds intelligent elevator control platforms for large commercial buildings across India.

Their software is deployed in:
- Corporate towers
- Shopping malls
- Airports
- Hospitals
- High-rise residential complexes

These buildings run **multiple elevators per building**, grouped into zones, handling thousands of passengers daily. The system must manage elevator assignments, floor requests, maintenance tracking, and ride logs efficiently.

---

##  System Architecture (Domain Rules)

- Each **building** can contain multiple **elevator shafts**
- Each **shaft** contains exactly **one elevator**
- Each **elevator** moves across a defined set of **floors**
- Elevators respond to **ride requests** generated by users from different floors

---

##  System Requirements

The platform must support:

| Capability | Description |
|---|---|
| Multi-building | Multiple buildings connected to the platform |
| Multi-elevator | Multiple elevators inside each building |
| Floor-level tracking | Requests tracked per floor |
| Ride allocation | Requests assigned to specific elevators |
| Status monitoring | Real-time elevator status tracking |
| Maintenance tracking | History of maintenance events per elevator |
| Usage history | Ride logs for analytics and reporting |

---

##  Design Goals

The schema should be able to answer operational questions like:

**Buildings & Floors**
- How many buildings are connected to the platform?
- Which floors belong to which building?

**Elevators**
- How many elevators exist inside a building?
- Which elevator serves which floors?
- Can multiple elevators serve the same floor?
- Can one elevator serve multiple floors?

**Status & Availability**
- What is the status of each elevator — idle, moving, or under maintenance?
- Can an elevator be temporarily disabled for maintenance?

**Requests & Rides**
- What requests were generated from which floors?
- Which elevator handled a request?
- Which requests are still pending?
- Which elevator handled the most requests?

**History & Analytics**
- How many rides did an elevator complete today?
- Can maintenance history be tracked per elevator?
- Can ride logs be recorded for analytics later?

##  ERD Scope — Entities to Design

Your ER diagram must include structures for:

```
1. Building
2. Floor
3. Elevator Shaft          ← optional but recommended
4. Elevator
5. Elevator ↔ Floor Map    ← many-to-many junction
6. Floor Request
7. Ride / Trip
8. Elevator Status
9. Maintenance Log
```

---

##  Key Relationships & Constraints

| Relationship | Cardinality | Notes |
|---|---|---|
| Building → Floors | One-to-many | A building has many floors |
| Building → Elevators | One-to-many | A building has many elevators |
| Elevator ↔ Floors | Many-to-many | Via a junction table |
| Floor → Requests | One-to-many | Users request from a floor |
| Request → Ride | One-to-one | One request yields one ride |
| Elevator → Rides | One-to-many | An elevator handles many rides |
| Elevator → Status | One-to-one | Separate from configuration data |
| Elevator → Maintenance Logs | One-to-many | History must never be overwritten |

---

##  Design Constraints

> These must be respected in your schema to ensure correctness and scalability.

- **Do not** store dynamic ride data (current floor, active trip) inside the Elevator entity
- **Do not** mix status fields with configuration/static data
- **Do not** overwrite maintenance records — each event must be a new row
- Elevator status (`idle`, `moving`, `maintenance`) must live in a **separate status table**
- The Elevator ↔ Floor relationship is **many-to-many** — model it with a junction table
